# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

A single [dev container feature](https://containers.dev/implementors/features/), `claude-code`, published to `ghcr.io/olibutzki/devcontainer-feature-claude-code/claude-code`. It installs Claude Code (native installer, no Node.js) inside a dev container and wires up everything needed to make that container a safe place to *opt into* `--dangerously-skip-permissions` for unattended work — without ever choosing that mode itself. See `README.md` for the full rationale and user-facing docs; this file is about working on the feature's implementation.

This is a 0.x, experimental project — breaking changes between versions are expected until 1.0.

## Commands

Requires Docker and the [devcontainer CLI](https://github.com/devcontainers/cli) (`npm install -g @devcontainers/cli`).

Run the full test suite (autogenerated `test.sh` + all scenarios in `scenarios.json`):

```bash
devcontainer features test -f claude-code -p . -i mcr.microsoft.com/devcontainers/base:ubuntu -u vscode
```

The `-i`/`-u` flags are required for the autogenerated `test.sh` run: without them the CLI defaults to `ubuntu:focal` as `root`, which this feature deliberately refuses to install into (see Architecture below). Scenarios in `scenarios.json` carry their own `image`/`remoteUser` and ignore `-i`/`-u`.

Useful variants:

```bash
# only the scenario tests (skip the slow autogenerated test.sh run)
devcontainer features test -f claude-code -p . --skip-autogenerated

# only tests whose name/scenario matches a substring
devcontainer features test -f claude-code -p . -i mcr.microsoft.com/devcontainers/base:ubuntu -u vscode --filter custom_allowed_domains
```

There's no separate lint/build step — the feature is plain shell + JSON, exercised entirely through `devcontainer features test`, which builds a real container from `src/claude-code` and runs the `check`/`reportResults` assertions in `test/claude-code/*.sh` against it.

To try it manually end-to-end (e.g. to check real browser-login persistence, which the automated tests can't do without credentials): point a throwaway project's `devcontainer.json` at `src/claude-code` via a local path `feature` reference, `devcontainer up`, then `devcontainer exec ... claude ...`.

## Architecture

### Install-time vs. runtime split

Everything happens in two phases, because Docker image builds and container starts are different contexts with different available information (env vars set on the *running* container, like `CLAUDE_CODE_OAUTH_TOKEN`, aren't visible during image build):

- **`src/claude-code/install.sh`** — runs once as root during image build. Installs OS packages, runs the Claude Code native installer *as the non-root remote user* (`su - "$_REMOTE_USER" -c ...`, since the installer writes to that user's `$HOME`), symlinks the binary to `/usr/local/bin/claude`, and drops the runtime scripts (below) plus their config into place. It does not touch anything that needs to react to runtime state.
- **`src/claude-code/scripts/claude-code-start.sh`** — the feature's `postStartCommand` (declared in `devcontainer-feature.json`, so it merges automatically with anything the consumer's own `devcontainer.json` defines — no user wiring required). Runs as the remote user on every container start, including after a rebuild. It re-applies the firewall (needs `sudo`, granted via a narrowly-scoped `NOPASSWD` sudoers entry install.sh writes) and conditionally runs the onboarding-fixup script based on a flag file (`/usr/local/etc/claude-code/auto-onboarding`) install.sh wrote from the `autoOnboarding` option.

### Firewall (`claude-code-init-firewall.sh`)

Default-deny iptables/ipset egress firewall for the *whole container*, not just Claude Code's traffic — rebuilt from scratch on every start (flush, rebuild ipset, re-resolve DNS, reject everything not in the allowlist). The allowlist is DNS-resolved at start time, not baked in as IPs, since most allowed hosts are on CDNs/anycast with rotating addresses. The domain list itself (defaults + the `allowedDomains` option, comma-split) is written by `install.sh` to `/usr/local/etc/claude-code/allowed-domains.txt` at build time and just read by the firewall script — keeps the shell script itself option-agnostic.

The default domain list is intentionally minimal: exactly the "Required for" domains from [Claude Code's network access requirements docs](https://code.claude.com/docs/en/network-config#network-access-requirements), nothing added by analogy (no GitHub, no npm/PyPI, no VS Code Server hosts — those are the *project's* concern and go through `allowedDomains`). Don't add domains to the default list without a documented Claude Code requirement backing them; that line gets crossed easily and quietly widens what every consumer's firewall allows by default.

### Persisted config volume

`devcontainer-feature.json` sets `containerEnv.CLAUDE_CONFIG_DIR=/home/.claude-code-config` and declares a `mounts` entry for a named volume `claude-code-config-${devcontainerId}` at that same path, directly in the feature (both `mounts` and `containerEnv` are valid top-level feature properties per the devcontainer spec) — so persistence works with zero devcontainer.json edits in consuming projects. `${devcontainerId}` is stable across rebuilds of the same dev container config/workspace and differs between projects/workspaces, which is what gives "same project, new container, still logged in" behavior.

The mount target is a fixed path, not the remote user's actual home directory, because `devcontainer-feature.json`'s `mounts` field is resolved before the remote user is known and can't reference the `_REMOTE_USER_HOME` value that's only available inside `install.sh`.

### Onboarding fixup (`claude-code-onboarding-init.sh`)

Sets `hasCompletedOnboarding: true` via `jq` (a safe, non-destructive merge — never overwrites the rest of the file) so a `CLAUDE_CODE_OAUTH_TOKEN`/API-key login doesn't hit Claude Code's interactive theme/setup screen. It patches **both** `$HOME/.claude.json` and `$CLAUDE_CONFIG_DIR/.claude.json` defensively: Claude Code's docs are ambiguous about whether `CLAUDE_CONFIG_DIR` relocates `~/.claude.json` (which holds the OAuth session) or only the `~/.claude` directory, so this covers both possibilities rather than guessing. If you ever confirm definitively which one Claude Code actually uses, this can be simplified to a single path.

### Non-root enforcement

`install.sh` hard-fails immediately if `$_REMOTE_USER` is `root`, with an explanatory message, rather than silently installing into a configuration that Claude Code itself would refuse to run `--dangerously-skip-permissions` in later. This is why `test.sh`/scenarios in `test/claude-code/` must always pass `-u <non-root-user>` (or set `remoteUser` in a scenario) — the CLI's default synthetic test container otherwise uses `root`.

### What this feature deliberately never does

No `permissions.defaultMode`, no `settings.json`, no wrapper around `claude` that injects `--dangerously-skip-permissions`. The feature's job is to make the container *safe* for that flag (non-root user + firewall + working auth), never to choose it. Keep new features/scripts consistent with this — anything that would make a permission mode automatic belongs in the consumer's own config, not here.

## Testing conventions

- `test/claude-code/test.sh` is the "default options" test, run automatically (the "autogenerated" test) against whatever image/user `-i`/`-u` specify.
- `test/claude-code/scenarios.json` defines named scenarios (currently `custom_allowed_domains`); each key needs a matching `test/claude-code/<name>.sh` script, and each scenario's own `image`/`remoteUser` (not the CLI's `-i`/`-u` flags) apply to it.
- Test scripts use the devcontainer CLI's `dev-container-features-test-lib` (`check "description" <command>` + `reportResults` at the end).
